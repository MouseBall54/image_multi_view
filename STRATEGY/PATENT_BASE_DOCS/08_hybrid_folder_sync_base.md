# 08. 하이브리드 폴더 동기화 (Electron watch + Web 메타 diff) (바탕 문서)

> 고지: 본 문서는 기술/전략 참고자료이며 법률 자문이 아닙니다.  
> 기준 문서: [PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md](../PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md)

## 문서 목적
웹/데스크톱 겸용 환경에서 폴더 변경 동기화를 어떻게 구현했는지 설명형으로 정리한다.  
핵심은 "플랫폼별 다른 감지 방식의 결합 + 변경분만 갱신"이다.

## 기능 정의
이 기능은 폴더 소스 유형에 따라 동기화 전략을 다르게 적용한다.

1. Electron 소스: `fs.watch` 이벤트 기반 감지 + 디바운스 갱신
2. Web FSA 소스: 주기적 메타데이터(`mtime`, `size`) diff 스캔
3. 변경 검출 시에만 파일 재로딩(선택적 I/O)

핵심 포인트:
1. 소스 타입 모델(`picker`, `electron`, `files`)
2. 이벤트 기반 경로와 폴링 기반 경로의 하이브리드 운영
3. 변경분만 반영하는 메타 중심 재스캔

## 사용방법
대표 시나리오:
1. Electron 환경에서 폴더를 로드하고 watch 등록 상태를 확인한다.
2. 폴더 내 파일을 수정/추가/삭제한다.
3. 앱에서 자동 반영 여부와 토스트 요약을 확인한다.
4. Web(FSA) 환경에서는 주기 스캔 간격 내 변경 반영 여부를 확인한다.

운영 팁:
1. 대용량 폴더는 전체 재로딩 대신 변경분 갱신 로그를 반드시 점검한다.
2. FSA 권한 만료 케이스를 별도 테스트해 예외 흐름을 문서화한다.

## 결과
정성 효과:
1. 플랫폼 제약이 달라도 일관된 "폴더 동기화 경험"을 제공한다.
2. 불필요한 파일 재읽기를 줄여 UI 응답성을 유지한다.

관찰 포인트(측정 후보):
1. 변경 반영 지연 시간(이벤트/폴링 경로별)
2. 변경 없음 상태에서 불필요 I/O 발생 횟수
3. 대량 폴더에서 갱신 시 UI 블로킹 시간

## 강점
1. 단일 방식 강제보다 실제 실행환경 특성에 맞춘 전략 선택이 가능하다.
2. 메타 diff 기반으로 성능과 정확성 균형을 맞출 수 있다.
3. Electron/Web 동작 차이를 내부에서 흡수해 사용자 혼란을 줄인다.
4. 실무 운영에서 자주 발생하는 추가/수정/삭제 시나리오를 안정적으로 다룬다.

## 특허 문서 전환을 위한 작성 포인트
권리화 핵심 구성요소:
1. 소스 유형 판별 단계
2. 이벤트 기반/폴링 기반 감지 전략 선택 단계
3. 메타 비교 후 선택적 파일 로딩 단계

차별 주장 포인트:
1. 플랫폼 차이를 숨기는 하이브리드 동기화 제어 구조
2. 변경분 중심 로딩으로 I/O를 줄이는 운영 효과

실험/증빙으로 보강할 요소:
1. 경로별 갱신 지연 시간 측정
2. 변경 파일 수 대비 I/O 호출 수 비교
3. 대량 폴더에서의 UI 응답성 로그

## 작성 시 주의사항/한계
1. 완전 실시간 보장은 환경 의존적이므로 디바운스/폴링 특성을 함께 명시해야 한다.
2. 파일시스템 권한 제약(Web FSA)은 기능 한계로 명확히 표기한다.
3. 특허 문서화 시 클라우드 동기화 일반론과 구분해 "런타임 분기 + 선택적 재로딩"을 중심으로 써야 한다.

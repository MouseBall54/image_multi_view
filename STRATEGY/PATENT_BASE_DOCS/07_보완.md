# 07. 컴퓨터 비전 필터 체인 실행·재현 파이프라인 (캐시·진행률·비용추정·JSON I/O·Python 변환 예정) (바탕 문서)

> 고지: 본 문서는 기술/전략 참고자료이며 법률 자문이 아닙니다.  
> 기준 문서: [PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md](../PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md)

## 문서 목적
compareX의 필터 체인 기능을 단순 시각 효과 UI가 아니라, **재현 가능한 컴퓨터 비전(CV) 이미지 처리 파이프라인**으로 정리한다.  
문서 목적은 특허 명세서 완성본 작성이 아니라, 특허 초안 작성자가 기술 과제-수단-효과를 빠르게 구조화할 수 있는 실무형 바탕 자료를 제공하는 것이다.

## 핵심 기술 한눈에 보기
1. 이 기술은 `filterType + params + enabled + 순서`를 하나의 실행 레시피로 취급해 동일 실험을 반복 재현한다.
2. 실행 엔진은 `cacheKey = imageHash + chainHash`를 사용해 동일 입력·동일 체인의 결과를 재사용하고, 진행률과 비용추정(estimatedMs)을 결합해 실행 가시성을 제공한다.
3. JSON import/export는 파라미터까지 포함한 round-trip 복원을 지원해 팀/환경 간 실험 재현성을 높인다.
4. Python(OpenCV/NumPy) 변환은 현재 구현이 아닌 `[예정 기능]`이며, 문서에는 설계 시안만 포함한다.

## 기능 정의
### 문제
기존 이미지 툴에서는 필터 조합 실험 시 다음 문제가 반복된다.
1. 동일 체인을 다시 실행해도 처리시간 예측이 어렵다.
2. 팀 공유 시 필터 순서나 파라미터가 누락되어 재현이 깨진다.
3. 웹에서 검증한 레시피를 코드 실험 환경으로 옮길 때 재작성 비용이 크다.

### 원리
1. 체인은 `enabled=true` 항목만 실행하고, 각 항목의 `filterType`과 `params`를 실행 단위로 본다.
2. 이미지 해시와 체인 해시를 결합한 키로 캐시를 조회해 히트/미스를 분기한다.
3. 파라미터는 임시 편집 상태에서 체인 아이템으로 스냅샷 저장되고, JSON `items[n].params`로 보존된다.
4. 비용추정은 필터별 기본비용, 파라미터 가중, 픽셀 수, 체인 길이를 결합한 모델로 제공된다.

### 효과
1. 반복 실행에서 체감 처리시간을 단축한다.
2. 레시피 공유 시 순서·파라미터 일치율을 높여 재현성을 확보한다.
3. 실행 전 예상 비용을 통해 실험 설계를 사전에 경량화할 수 있다.

## [현재 구현됨] 처리 파이프라인 개요
1. 체인에서 `enabled=true` 아이템만 추출한다.
2. `imageHash`와 `chainHash`를 결합해 `cacheKey`를 생성한다.
3. 캐시 히트 시 결과를 복원하고 진행률 피드백을 유지한다.
4. 캐시 미스 시 필터를 순차 적용하며 단계별 진행률을 갱신한다.
5. 최종 결과를 캐시에 저장하고 크기 상한 정책으로 정리한다.
6. 실행 전 `estimateFilterChainCost`로 `estimatedMs`를 계산한다.

사용자 관찰값과 기술효과(1:1 매핑):
1. 관찰값: 같은 체인을 다시 돌리면 빠르다.  
기술효과: 캐시 재사용으로 재실행 지연 감소.
2. 관찰값: 긴 체인에서도 진행 상태가 끊기지 않는다.  
기술효과: 단계 기반 progress callback으로 상태 가시성 유지.
3. 관찰값: 실행 전에 대략 시간을 보고 체인을 조정할 수 있다.  
기술효과: 사전 비용추정으로 탐색/튜닝 비용 절감.

## [현재 구현됨] 파라미터 조절 및 저장 원리
핵심 데이터 흐름(고정):

```text
UI 슬라이더/숫자입력
  -> tempViewerFilterParams
  -> addToFilterCart / updateFilterCartItem
  -> filterCart[n].params
  -> exportFilterChain
  -> JSON items[n].params
```

단계 설명:
1. 사용자가 Filter Editor에서 파라미터를 조절하면 `tempViewerFilterParams`가 즉시 갱신된다.
2. 체인에 추가할 때(`addToFilterCart`) 해당 시점 파라미터가 아이템 `params`로 스냅샷 저장된다.
3. 체인 아이템 재편집 시(`updateFilterCartItem`) 아이템별 `params`가 갱신된다.
4. export 시 필터별 유효 파라미터만 추려 JSON으로 저장된다.

기술 의미:
1. “필터 종류는 같지만 값이 달라 결과가 달라지는 문제”를 구조적으로 줄인다.
2. 체인 공유 시 파라미터까지 동일하게 전달되어 재현 일관성이 높아진다.

## [현재 구현됨] JSON import/export 기능
### 1) JSON 구조 원리
이 포맷은 필터 체인을 앱 내부 구현과 분리한 **플랫폼 독립 레시피 포맷**이다.
1. 기본 구조: `name`, `items[]`
2. 아이템 구조: `id`, `filterType`, `params`, `enabled`
3. wrapper 구조: `version`, `exportedAt`, `chain`, `metadata`

### 2) 저장(export) 원리
`FILTER_PARAMETER_MAP`으로 필터 타입별 유효 파라미터만 정규화해 저장한다.
1. 장점: 불필요 필드 제거로 가독성/호환성 개선
2. 기술 의미: 체인 교환 시 파라미터 스키마를 안정화

간단 예시(direct):
```json
{
  "name": "basic-denoise-edge",
  "items": [
    {
      "id": "filter-1",
      "filterType": "gaussianblur",
      "params": { "kernelSize": 5, "sigma": 1.2 },
      "enabled": true
    },
    {
      "id": "filter-2",
      "filterType": "canny",
      "params": { "lowThreshold": 60, "highThreshold": 150 },
      "enabled": true
    }
  ]
}
```

실전 예시(wrapper):
```json
{
  "version": "1.0.0",
  "exportedAt": 1767139200000,
  "chain": {
    "id": "exported-1767139200000",
    "name": "inspection-chain",
    "items": [
      {
        "id": "filter-abc",
        "filterType": "clahe",
        "params": { "clipLimit": 2.5, "gridSize": 8 },
        "enabled": true
      },
      {
        "id": "filter-def",
        "filterType": "threshold_adaptive_gaussian",
        "params": { "blockSize": 21, "constant": 3, "maxValue": 255, "binaryInvert": false },
        "enabled": true
      }
    ],
    "createdAt": 1767139100000,
    "modifiedAt": 1767139200000
  },
  "metadata": {
    "appVersion": "1.0.0",
    "description": "Filter chain: inspection-chain"
  }
}
```

### 3) 불러오기(import) 원리
import는 아래 3개 포맷을 순차 수용한다.
1. `version + chain` wrapper
2. `name + items` direct
3. `preset.name + preset.chain` preset

이후 구조 검증:
1. 체인 레벨: `name` 문자열, `items` 배열
2. 아이템 레벨: `id/filterType/params/enabled` 타입 검사

### 4) 실패 시 피드백
1. JSON 파싱 오류/필드 오류를 명시적 메시지로 반환한다.
2. UI 토스트로 실패 원인과 재시도 포인트를 안내한다.

## [예정 기능] Python 코드 변환/내보내기
### 목적
웹 앱에서 검증한 체인을 Python(OpenCV/NumPy) 코드로 전개해 오프라인 실험 파이프라인에서 직접 실행 가능하도록 한다.

### 출력물(예정)
1. `chain_name.py`: 체인 순회 실행 스크립트
2. `chain_name.mapping.json`(선택): 필터 타입-함수 매핑 로그

### JSON -> Python 매핑 시안
| filterType | Python 매핑 예 |
| --- | --- |
| `gaussianblur` | `cv2.GaussianBlur(img, (k, k), sigma)` |
| `median` | `cv2.medianBlur(img, k)` |
| `canny` | `cv2.Canny(gray, low, high)` |
| `threshold_binary` | `cv2.threshold(gray, t, maxv, mode)` |
| `threshold_adaptive_*` | `cv2.adaptiveThreshold(...)` |
| `histogramequalization` | `cv2.equalizeHist(gray)` |
| `clahe` | `cv2.createCLAHE(...).apply(gray)` |
| `morph_*` | `cv2.morphologyEx(..., kernel, iterations=...)` |

상태 경계:
1. 본 항목은 `[예정 기능]`이며 현재 저장소에는 미구현이다.
2. 현재 앱에서 즉시 Python export가 가능하다는 의미로 해석하면 안 된다.

## 적용 로직/식(구현 기준)
### 1) 캐시 키
```text
chainHash = join("|", enabledItems.map(item => filterType + "-" + JSON.stringify(params)))
cacheKey  = imageHash + "-" + chainHash
```
해석: 같은 입력 이미지와 같은 활성 체인이면 동일 키를 생성한다.

### 2) 이미지 해시
```text
imageHash = "{width}x{height}-{first_10_pixels_subset}"
```
해석: 전체 픽셀 해시보다 빠르게 동일 입력 여부를 판별한다.

### 3) 비용추정
```text
pixelCount = imageWidth * imageHeight
totalCost = Σ(baseCost(filterType) * paramMultiplier)
estimatedMs = (totalCost * pixelCount / 1000) + (enabledItems.length * 50)
```
해석: 필터 종류·파라미터·해상도·체인 길이를 결합해 예상 시간을 계산한다.

파라미터 가중 예:
```text
if kernelSize > 5:
  paramMultiplier *= (kernelSize / 5)^2
```

모델 한계(현재 구현):
1. 선형 근사 중심이라 장치/브라우저/데이터 분포별 오차가 발생한다.
2. 필터 상호작용(앞단 처리 결과가 뒷단 복잡도에 미치는 영향)을 완전 반영하지 않는다.

### 등급 상향을 위한 모델 보강 설계(문서상 제안)
`[예정 보강 아이디어]`
1. 실행 로그 기반 보정: 실제 실행시간 로그로 `baseCost`를 주기적으로 재보정.
2. 해상도 구간별 보정식: 저해상도/고해상도 구간별 계수 분리.
3. 체인 상호작용 보정: 특정 필터 조합 빈도 기반 correction factor 추가.

## 사용방법
### 시나리오 A: 캐시 히트 기반 성능 실험
1. 동일 이미지 1장을 선택하고 3~6단계 체인을 구성한다.
2. 1차 실행(캐시 미스) 시간을 기록한다.
3. 같은 체인을 2차 실행(캐시 히트)하고 시간을 비교한다.
4. 실행 전 `estimatedMs`와 실측시간 차이를 기록한다.

확인할 관찰값:
1. 1차/2차 처리시간(ms)
2. 캐시 히트 체감 응답성
3. 추정 대비 실측 오차율

### 시나리오 B: JSON round-trip 재현 실험
1. 필터 2~3개의 파라미터를 조절해 체인을 만든다.
2. JSON export 후 다른 세션/환경에서 import 한다.
3. `filterType/params/enabled`가 동일하게 복원되는지 확인한다.
4. 결과 이미지를 비교해 재현 일관성을 확인한다.

확인할 관찰값:
1. 구조 일치율(필드 비교)
2. 시각 비교 또는 픽셀 차이 통계
3. import 실패 유형별 에러 메시지 적합성

## 결과
### 정성 효과
1. 필터 기능이 “단발 효과”에서 “재현 파이프라인”으로 전환된다.
2. 실행 성능과 실험 공유가 한 구조로 연결된다.
3. 향후 Python 연계 설계의 기반 포맷(JSON)을 선확보한다.

### 정량 관찰 항목
1. 캐시 히트 시간절감률
2. `estimatedMs` 대비 실제시간 오차율
3. JSON round-trip 일치율(`filterType/params/enabled`)
4. 결과 이미지 차이 통계(예: 픽셀 차이율)

### 측정 방법 예시
1. 이미지 10장 x 체인 5종 x 3회 반복
2. 평균/표준편차 및 95% 신뢰구간 기록
3. round-trip 전후 필드 diff + 결과 이미지 diff 동시 측정

## 강점
1. 실행-예측-재현을 분리하지 않고 하나의 작업 루프로 제공한다.
2. 캐시/진행률/비용추정을 결합해 사용자 의사결정 속도를 높인다.
3. 파라미터 단위 저장으로 팀 재현성 리스크를 낮춘다.
4. 81개 CV 필터 조합 실험을 체인 레시피 단위로 관리할 수 있다.
5. JSON 포맷이 향후 코드 변환(OpenCV/NumPy) 확장의 중간 계층이 된다.

## 특허성 관점 요약 (압축)
1. 선행기술군 경향: 캐시/진행률/비용추정/입출력은 일반 시스템·DB 영역에서 공지성이 높다.
2. 본 기술 차별축: CV 필터 체인 도메인에서 실행·가시성·재현(JSON)을 사용자 실험 파이프라인으로 통합했다.
3. 신규성/진보성 관점: 조합 수준 신규성은 중간 이상 잠재력이 있으나, 진보성은 청구항 설계(범위 설정)에 크게 의존한다.
4. 리스크: 캐시/비용추정/JSON I/O를 일반 개념으로 넓게 쓰면 자명성 지적을 받을 가능성이 높다.
5. 강화 포인트: 정량 실험 데이터, 일반화된 구성 표현, 도메인별 실시예(반도체/의료/OCR)를 함께 제시해야 한다.

function generateCacheKey(imageHash: string, chainItems: FilterChainItem[]): string {
  const chainHash = chainItems
    .filter(item => item.enabled)
    .map(item => `${item.filterType}-${JSON.stringify(item.params)}`)
    .join('|');
  return `${imageHash}-${chainHash}`;
}

## 특허 문서 전환을 위한 작성 포인트
### 1) 기술과제 1문장 템플릿
“컴퓨터 비전 필터 체인에 대해, 실행 전후/중 비용·진행률·캐시 재사용 정보를 제공하고, JSON 기반 체인 교환을 통해 다환경 재현성을 확보하는 실행·재현 파이프라인을 제공한다.”

### 2) 독립항 구성요소 템플릿
1. 필터 체인 정의부(`filterType`, `params`, `enabled`, 순서)
2. 캐시 키 생성/조회부(`imageHash + chainHash`)
3. 체인 실행 제어부(히트/미스 분기)
4. 진행률 출력부(progress callback)
5. 비용추정부(필터별 비용 + 파라미터 가중 + 해상도)
6. 레시피 입출력부(JSON import/export + 검증)

### 3) 종속항 후보(6개)
1. 캐시 히트 시 진행률 시뮬레이션을 유지하는 구성
2. 필터 타입별 유효 파라미터 정규화(FILTER_PARAMETER_MAP) 구성
3. 다중 JSON 포맷(wrapper/direct/preset) 수용 구성
4. 파라미터 스냅샷 저장(`tempViewerFilterParams -> filterCart.params`) 구성
5. 커널 크기 기반 파라미터 가중 비용추정 구성
6. 캐시 상한 및 정리 정책(MAX_CACHE_SIZE 기반) 구성

### 4) 실험증빙 패키지(체인 도메인 지표)
1. 캐시 히트 시간절감률
2. `estimatedMs` 대비 실제시간 오차율
3. JSON round-trip 일치율(`filterType/params/enabled`)
4. 결과 이미지 차이 통계(픽셀 차이율/분포)

## 작성 시 주의사항/한계
1. `[예정 기능]`은 구현 완료처럼 서술하지 않는다.
2. 비용추정은 안내값이며 환경/브라우저/데이터에 따라 오차가 발생한다.
3. 필터 개수(현재 81개)는 코드 변경 시 달라질 수 있으므로 문서 갱신 시 재검증한다.
4. JSON 스키마가 바뀌면 예시와 검증 로직 설명을 함께 갱신한다.
5. 본 기능은 사용자 정의 체인 기반이며 자동 최적 체인 탐색을 보장하지 않는다.
6. 특허 문구에서는 일반 개념 과대청구를 피하고, CV 체인 도메인 특화 구성으로 범위를 정교화해야 한다.

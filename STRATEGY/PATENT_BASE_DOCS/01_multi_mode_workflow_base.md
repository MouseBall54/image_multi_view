# 01. 멀티모드 통합 비교 워크플로우 전환 구조 (바탕 문서)

> 고지: 본 문서는 기술/전략 참고자료이며 법률 자문이 아닙니다.  
> 기준 문서: [PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md](../PATENT_PORTFOLIO_ANALYSIS_KR_PCT.md)

## 문서 목적
compareX의 Compare/Pinpoint/Analysis 전환 구조를 "설명 가능한 기술 문서" 형태로 정리한다.  
핵심은 UI 탭 소개가 아니라, 모드 전환 시 상태를 어떻게 유지/리셋하는지에 대한 구현 논리다.

## 기능 정의
이 기능은 하나의 전역 상태 저장소를 중심으로 3개 모드를 운영한다.  
모드 전환이 발생하면 사전에 정의된 상태전이 정책에 따라 일부 상태를 유지하고 일부 상태를 초기화한다.

핵심 포인트:
1. 공통 상태 저장소 기반 모드 운영
2. 모드 전환 이벤트 감지 후 정책 기반 상태 재구성
3. 사용자 입력 경로(파일 로딩, 뷰어 조작, 캡처)를 공통 인터페이스로 유지

## 사용방법
대표 시나리오:
1. Compare 모드에서 이미지 폴더를 로드하고 뷰어 레이아웃을 구성한다.
2. 상단 모드 스위처로 Pinpoint 모드로 전환한다.
3. 전환 직후 뷰포트/선택 상태가 정책대로 리셋되었는지, 파일 기반 작업 흐름은 유지되는지 확인한다.
4. Analysis 모드로 재전환해 동일 입력 자산을 기반으로 필터 실험을 이어간다.

운영 팁:
1. "완전 유지"가 아니라 "선택적 유지" 설계라는 점을 사용자 가이드에 명시한다.
2. 모드 전환 전/후 상태를 로그로 기록하면 검증 및 문서화가 쉬워진다.

## 결과
정성 효과:
1. 모드 간 작업 목적이 달라도 하나의 앱 안에서 연속 작업이 가능하다.
2. 도구 전환에 따른 재학습/재설정 비용이 줄어든다.

관찰 포인트(측정 후보):
1. 모드 전환 소요 시간(클릭부터 인터랙션 가능 시점까지)
2. 전환 후 사용자가 수동 재설정해야 하는 항목 수
3. 동일 업무를 분리 툴로 수행할 때 대비 단계 수 감소율

## 강점
1. 기능 확장 시 모드별 기능을 추가하되 상태 아키텍처는 재사용 가능하다.
2. 사용자 관점에서 워크플로우 단절이 줄어들어 생산성이 높다.
3. 상태전이 규칙을 명시적으로 관리하므로 QA 범위를 체계화할 수 있다.
4. 기업 도입 시 운영 절차 표준화(교육/매뉴얼) 작성이 쉽다.

## 특허 문서 전환을 위한 작성 포인트
권리화 핵심 구성요소:
1. 공통 전역 상태 저장소
2. 모드 전환 이벤트 처리기
3. 상태 유지/초기화 정책 테이블

차별 주장 포인트:
1. 단순 탭 전환이 아니라 상태전이 정책 기반 연속 작업 구조
2. 모드별 독립 앱 대비 재설정 단계 감소

실험/증빙으로 보강할 요소:
1. 사용자 테스트 기반 전환 시간/오류율 비교
2. 모드 전환 후 재설정 항목 수 통계

## 작성 시 주의사항/한계
1. 실제 구현은 일부 상태를 의도적으로 초기화하므로 "모든 상태 유지"로 서술하면 안 된다.
2. UI 명칭 중심 설명보다 상태전이 메커니즘 중심으로 정리해야 한다.
3. 특허화 시 "워크플로우 일반론"으로 넓게 쓰면 선행기술 충돌 가능성이 커진다.
